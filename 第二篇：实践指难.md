#### 3、Skills 版

**这个方案其实已经很不错了，但我们可以想一个问题来探索点更多的东西，以实际工作场景为例：比如正式员工这个星期从领导那儿接了个 App 开发的活（正式员工就像 agent），我们每天安排得满满当当，每天一上班脑袋里就猛猛加载开发 App 用的技术（类似于 CLAUDE.md），其它技术我们暂时用不到就没往脑袋里加载。但是周三领导突然跟我们说你们先开发几个后台的接口、这个比较紧急，于是我们就赶紧往脑袋里加载开发后台的技术（类似于 Skills），哐哐一顿干，干倒是干完了，但却影响了我们本来的主线任务。于是我们就思考如果这个后台任务不是我们自己做，而是招个专门搞后台的实习生来干那不就不会影响我们原来的主线任务了吗。的确如此，Subagent 就像实习生，而我们就像那个主 agent，也就是说正式员工多技术栈能干的事也可以找个实习生来干，即主 agent skills 能干的事也可以搞个 subagent 来干，至于它俩的区别和适用场景我们稍后总结，现在先让 subagent 干干活再说。**

#### 4、Subagents 版

**Subagents 的特点：1、异步：主 agent 在执行任务的期间，可以启动一个或多个子 agent 异步地去执行它们擅长的特定任务，它们执行完任务后会向主 agent 汇报结果，这样一来就不是主 agent 同步地一个一个任务执行了，任务执行效率就提高了。2、独立上下文：subagent 有自己完全独立的上下文，启动 subagent 的时候，主 agent 会将自己的上下文压缩后传递给 subagent（就像我们把当前对项目的所有理解压缩到一篇文档里交给实习生，后续实习生就会基于这篇交接文档作为项目背景干活了），这就是 subagent 接收的信息，然后 subagent 执行任务期间所产生的上下文信息绝不会回传给主 agent、而仅仅是增加子 agent 自己的上下文（就像实习生有自己的每日安排），等subagent干完活以后它又会把自己的上下文压缩会回传给主 agent、主 agent 就拿到这个任务的最终执行结果了（就像实习生给我们上报这个活最终干得咋样了），这样一来主 agent 的上下文就不会受太大影响、只会加入 subagent 执行结果这么一点上下文（就像我们只需要知道结果就行了，中间他具体咋干的我们不需要费精力关心，我们的进度也就不会耽误）。3、生命周期：subagent 的生命周期是很短的，仅存在于一轮对话的一个任务中，任务一旦完成它就是释放了，下轮对话如何还需要它，那就重新创建，跟实习生的实习期类似。**

###### skills 和 subagent 区别与选择

* skills 可以看作是一个人所具备的技能、技能没有自主性，而 subagent 则是一个人、subagent 是自主的、也就是说它有自己的三件套【神经系统⚡️+大脑🧠+工具🛠️】
* skills 会全量影响主 agent 的上下文，因为 skills 就是在主 agent 上运行的，subagent 则几乎不影响主 agent 的上下文，可以类比后台那个活如果让 App 开发用后台技术自己做那肯定会猛猛浪费 App 开发的精力（精力类似于上下文），而如果 App 开发把后台任务找一个实习生做，那就几乎不浪费 App 开发的精力
* 那选择 skill 还是 subagent？1、我们可以直接以实际工作中的岗位来搞，比如项目经理、产品经理、开发、测试啥的，这都可以给它搞个 agent，这绝对不能是 skill，因为既然都谈到岗位了，那就代表这项任务有一定的复杂度，执行期间必然会产生大量上下文信息（像搜索大量文件、建立这些文件之间的连接关系、进行深度思考这些事都会产生大量上下文信息），这如果用 skills 的话，主 agent 的上下文很快就被撑爆。2、我们还可以按一个岗位工作流的不同阶段来搞，比如写 UI 的 agent、对接口的 agent、把数据和 UI 打通的 agent、改 bug 的 agent、codereview 的 agent、发版的 agent 等等，因为这每个环节要干的事也是一个小系统不是什么散活，也会产生大量中间上下文。3、一些散活就搞成 skill 吧，毕竟这些散活对主 agent 的上下文影响有限。

#### 5、Plugin 版

Plugin 是把 mcp servers（可单独分发）、hooks、subagents、skills（可单独分发） 一整套能力打包起来，分发给别人共享使用，别人一键安装即可让他们的 CC 获得一整套能力。

假设这两个 agent 我自己用的挺好，想分发给别人共享使用，那就可以打包成一个 plugin

######  打包流程

* **1. 创建 plugin 目录结构**

```
 my-purchase-plugin/                                                  
 ├── plugin.json     # plugin 元数据/清单                                      
 ├── agents/                                                       
 │  ├── merge-quotes.md # 你的合并报价表 agent                                     
 │  └── calc-price.md  # 你的计算售价 agent                                      
 ├── skills/       # 可选，slash commands
 ├── mcp-servers/  # 可选，工具
 └── hooks/        # 可选，自动化钩子
```

* **2. 编写** **plugin.json**

```json
{
  "name": "purchase-quote-tools",
  "version": "1.0.0",
  "description": "采购报价表处理工具：合并报价表 + 计算售价",
  "author": "你的名字",
  "components": {
    "agents": [
      "agents/merge-quotes.md",
      "agents/calc-price.md"
    ]
  }
}
```

* **3. 分发方式**

```
- 本地目录：直接把整个目录发给别人（我们演示下本地）
- Git 仓库：推到 GitHub，别人从 URL 安装
- Plugin 市场/registry（社区在建
```

* **4. 搞个本地本地插件市场**

plugin install 只支持从 marketplace 安装，不支持直接指定本地路径。所以本地路径插件需要先把所有插件的共同父目录注册为 marketplace，再安装，流程是：

```json
// 在 所有插件的共同父目录下创建一个 .claude-plugin 目录，在 .claude-plugin 目录下创建一个 marketplace.json 文件，填写内容：
{
  "name": "my-plugins",
  "owner": {
    "name": "local"
  },
  "plugins": [
    {
      "name": "my-purchase-plugin",
      "source": "./my-purchase-plugin"
    }
  ]
}

// 配好之后启动 claude，输入 /plugins - marketplaces 应该能看到本地市场
```

###### **安装流程（使用者侧）**

```bash
# 启动 claude 后，从本地目录安装                          
plugin install ${plugin 的绝对路径} # 默认把 plugin 安装到用户级
plugin install --scope user ${plugin 的绝对路径} # 同上
plugin install --scope project ${plugin 的绝对路径} # 项目级，可提交 git
plugin install --scope local ${plugin 的绝对路径} # 项目级，不可提交 git
```

安装 plugin 后，其实就把打包在其中的 mcp servers、hooks、subagents、skills 给拿出来再放到到相应的目录，就好像被分享者自己写的这些东西一样，可以愉快地使用了。
