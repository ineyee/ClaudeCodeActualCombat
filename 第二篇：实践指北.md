#### 2、CLAUDE.md 版

**CLAUDE.md 的特点：每次打开会话就会加载，并且后续当前会话里的每轮对话也都会把 CLAUDE.md 作为上下文带进去。这样一来，我们每次就只需要发送一个“合并价格表”就可以了，重复的提示词就不用再复制了，CLAUDE.md 就像给 CC 添加了的记忆，不用我们一直重复跟它讲**

###### 要加新需求了

现在我们除了合并报价表的需求，还需要计算出每张报价表里每个价格上浮 15% 利润后的价格，这个价格就是平台售价。于是我们又得教 CC 怎么干这么活了。同样地为了避免每次干这个活都得复制一大堆提示词，我们依然直接把这个新需求的提示词直接放进 CLAUDE.MD 里

```markdown
## 计算售价流程
当用户表达以下“意图”时，执行【计算售价流程】：
- 计算售价
- 搞下利润价
- 或任何语义上等价的表达

⚠️ 注意：
- 不要求用户必须说固定关键词
- 必须基于“语义理解”判断
- 如果不确定，先询问用户确认再执行

1. 第一步：从”每日个人报价表“目录里任选一份文档，打开此文档，只修改有数据的”价格“列，没数据的”价格“列不用管
2. 第二步：怎么修改”价格“列？如果某个单元格里没数据或者没有数字（比如是纯汉字），那就不用处理该单元格；否则就拿出其中的数字，除以 0.85，得到一个值，除不尽的话就四舍五入保留两位小数，用括号（）把这个值括起来拼接在原数据后面即可
3. 第三步：再选另外一份文档，重复第一步和第二步，确保每份文档都处理但是又不重复处理

## 计算售价示例
比如有一个单元格里是纯数字 ”3.25“，那就 3.25 除以 0.85 约等于 3.82，于是这个单元格的最终新值就是 ”3.25（3.82）“；又比如一个单元格里既有数字也有其它非数字字符 ”35特价“，那就把数字 35 截取出来，35 除以 0.85 约等于 41.18，于是这个单元格的最终新值就是 ”35特价（41.18）“
```

**这个方案有点问题：随着需求越增越多，CC 每次执行任务前需要遵守的规则就会越来越多，同样为了避免我们每次执行任务需要复制一大堆上下文，那么新增的规则就会都写在 CLAUDE.md 里，那 CLAUDE.md 势必会变得越来越臃肿，这就会带来两个问题：1、一个会话才刚启动，都还没开始干活，CLAUDE.md 就已经占据了大量上下文，隐形 token 消耗量暴涨；2、CLAUDE.md 里面的规则太多了，CC 执行一项特定的任务时就会读取到太多无关规则，这会导致 CC 变傻变慢，类似于“我读书太多，学杂了”。Skills 是个办法，我们可以把 CLAUDE.md 里只属于特定任务的规则分别抽取出来放到单独的 skill 里面去。**

#### 3、Skills 版

**Skills 的特点：每次打开会话，肯定会完整加载 CLAUDE.md，但是只会加载所有 skills 的 name 和 description 这些 skills 的元信息、而不是所有的 skills 的完整文档，也就是 CLAUDE.md + skills 元信息列表，然后后续当前会话里的每轮对话也都会把 CLAUDE.md 作为上下文带进去，一旦某轮对话中模型判断到需要用某个 skill 来处理当前任务才回去加载指定 skill 的完整文档。换句话说 skills 是按需加载、渐进式披露，就可以完美解决这两个问题，这是 skills 跟 CLAUDE.md 的区别。但两者都还是 CC 的记忆，只不过 CLAUDE.md 是会话启动就完整载入大脑，skills 是需要回忆时才回忆起。**

###### 一个好的 skill 模板

**讨论、优化，这也是 CC 给的建议，基本跟我们的提示词一致，感觉还是我们用户提示词那些内容**

```markdown
// skill 的元数据部分（主要负责简述当前 skill 是什么、能干啥）
// name 和 description 必填，其它 4 项可选
---
name: skill-name                        // skill 名称，见名知意，最好数字字母中划线，1-64 字符
description: 什么时候用这个 skill         // skill 功能描述和触发条件，非常关键，1-1024 字符，触发条件同样表明”意图触发“而不是”关键词“触发

disable-model-invocation: false         // 可选：是否禁止 Claude 自动触发、只能由用户调用，默认 false、即允许 Claude 自动触发
user-invocable: false                   // 可选：是否允许用户通过 /skill-name 调用，默认 true
allowed-tools: Read, Grep, Glob, Bash   // 可选：限制可用工具（根据实际场景从 CC 内置工具里选），默认值 ALL
context: fork                           // 可选：是否自动开启一个 subagent 来加载当前 skill 然后去执行任务，默认 inline、代表就让主 agent 来加载当前 skill 然后去执行任务，这样一来 skill 就会放在主 agent 的上下文里；而如果是 fork，那就会自动开启一个 subagent 来加载当前 skill 然后去执行任务，那 skill 就会放在 subagent 的独立上下文里，不污染主 agent 的上下文
---
```
